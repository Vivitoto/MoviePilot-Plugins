# 规则模式（第一版）

这部分沉淀历史 SQL 中反复出现的业务规则模式。它比纯语法更重要。

## 1. 剔除规则高频出现

常见剔除对象：
- 行业卡
- 物联网
- 企业证件入网
- 亲副卡 / 主副卡群组
- 校园 / 沃派 / eSIM / 小号
- CPE / 国际类 / 一卡多终端 / 一卡多卡
- 特定套餐清单
- 风险客群
- 黑名单

生成 SQL 时，如果任务是外呼、升套、维系名单，默认要考虑是否存在剔除规则。

## 2. 保留规则也很关键

常见保留条件：
- 在网
- 正常开机
- 入网时长达到某门槛
- 上月有使用量 / 有出账
- 归入市场部统一维系
- 指定 tele_type / 指定用户类型

## 3. 时间口径常按月迭代

这套 SQL 很依赖月末/日末口径：
- `month_id`, `day_id`
- `to_char(sysdate - 2, 'YYYYMM')`
- `to_char(last_day(...), 'dd')`
- `add_months(...)`

生成时要注意：
- 不要把自然语言里的“上月”“当月”“次月”简单翻译成固定日期
- 优先沿用历史脚本里的月末、T+1、次月追踪写法

## 4. 合约与升套规则常变化

历史脚本里常见：
- 按月份修改入网时长门槛
- 按档位区分策略
- 合约到期限制不同
- 倍享 / 权益 / 低消 / 升 20 送 20 等不同托底逻辑

因此如果用户没指定版本，默认：
- 先采用最近、最完整、注释最充分的脚本风格
- 对明显依赖月份口径的规则，必要时提醒用户确认

## 5. 输出字段常带业务标签

名单型 SQL 里常见输出：
- 合约 ID / 名称 / 开始时间 / 到期时间
- 是否新办终端合约
- 是否新办移宽融合
- 是否办理 FTTR
- 当前状态 / 销户时间 / 欠费金额
- 入网月 / 入网渠道 / 累计充值
- 推荐策略 / 是否特邀低消 / 是否融合用户

生成 SQL 时，不要只输出 user_id；优先考虑可直接用于业务执行的字段集合。

## 6. 注释是规则来源，不是装饰

这批 SQL 中大量关键业务口径只写在注释里。处理方式：
- 优先相信注释表达的业务条件
- 再检查 SQL 是否完全体现
- 如果注释与 SQL 条件可能不一致，优先提醒用户确认，而不是默默忽略
